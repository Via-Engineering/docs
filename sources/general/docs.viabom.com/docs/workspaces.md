# Workspaces

Copy page

Understand how Via scopes shared projects, libraries, rulesets, and settings to the active organization workspace.

Workspaces are the organization-level container for shared work in Via.

## [What A Workspace Controls](https://docs.viabom.com/#what-a-workspace-controls)

Your active workspace determines which shared items you are looking at, such as:

- projects
- libraries
- rulesets
- workspace settings and membership

If a project or ruleset seems to have vanished, the active workspace is the first thing to check.

## [Switching Workspaces](https://docs.viabom.com/#switching-workspaces)

Use the workspace selector at the top of the main sidebar to move between organizations.

That menu is also where you can:

- search your available workspaces
- create a workspace
- join a workspace
- open workspace settings

## [Creating And Joining](https://docs.viabom.com/#creating-and-joining)

Create a new workspace when you need a separate shared environment for a team, customer, or product line.

Join an existing workspace when someone has invited you into one. If the right workspace is not showing up, confirm the invitation reached the same email you use to sign in.

## [Shared Vs Personal Expectations](https://docs.viabom.com/#shared-vs-personal-expectations)

Workspaces are meant for shared context, not just personal organization. That means other members can see and reuse the projects, libraries, and rulesets that live there, subject to the product's current permissions model.

## [Workspace Settings](https://docs.viabom.com/#workspace-settings)

Workspace settings are where Via exposes organization-level setup like:

- profile and branding
- members and invitations
- domain-related settings

You generally do not need these settings for day-to-day part work, but they are important when a team is getting set up.

## [Current Notes](https://docs.viabom.com/#current-notes)

- workspace behavior is already central to the product, so stay aware of which organization is active before searching or editing shared assets
- when something looks wrong, include the workspace name in bug reports
